# Dashboard Context

Last updated: 2026-04-13

## Product Intent

The dashboard is a mobile-first visual map of assets moving through BOB Gateway.

It should feel like a live route radar:

- BTC, WBTC, stablecoins, and native assets are the moving objects.
- BOB Gateway is the central hub.
- BOB Mainnet is one of the connected chains, separate from BOB Gateway.
- Supported chains are arranged around BOB Gateway.
- Animated pulses show both the route and the asset family.
- Fading trails show recent traces after a fast quote/route event has passed.
- Updates, gas, and native prices are supporting context, not the main story.

Do not turn this screen back into an operator status table. The user rejected that shape.

## User-Facing Copy

Keep copy simple and visual.

Good:

- `Gateway Flow Map`
- `BTC -> BOB -> 8개 체인`
- `운영 상태`
- `전략 지도`
- `Recent Activity`
- `Asset Coverage`
- `가스와 가격`
- `새 업데이트 없음`

Avoid:

- internal module names
- schema jargon
- signer/executor details
- long audit explanations
- strategy or profitability claims

The dashboard may mention blockers only as plain-language state, for example `관찰 데이터 축적 중`.

`운영 상태` 카드는 공식 live 상태, 현재 prelive 단계, active blocker, 그리고 "실험 흔적 != 공식 live 승인" 경계를 짧게 설명할 수 있다.

`전략 지도` 카드는 native BTC가 여러 자산 레일과 allocator 후보로 분기된다는 사실, 그리고 `promotable` / `allocation-ready` / `review-only` 차이를 plain-language 요약으로 보여줄 수 있다.

## Safety Boundary

The dashboard is read-only.

It must never:

- hold private keys
- expose wallet seed material
- sign transactions
- decide whether to trade
- claim live trading is enabled
- include execution permission

`liveTrading` reflects whether the runtime gates (audit, gateway probes, gas freshness) currently pass. `ALLOWED` is a normal state, not an exceptional one — it does not mean the dashboard decides to trade. The dashboard still must not hold keys, sign transactions, or execute; it only reports the gate state.

Cloudflare is only allowed to serve static files from `dashboard/public`.

## Data Flow

Source JSONL files live under `data/`.

The dashboard does not read those raw JSONL files directly. It reads:

- `dashboard/public/dashboard-status.json`

That file is generated by:

```bash
npm run status:dashboard
```

The generator also writes:

- `data/dashboard-status.json`

`data/dashboard-status.json` is the local copy. `dashboard/public/dashboard-status.json` is the read-only public copy.

## Important Files

- `src/status/dashboard-status.mjs`: builds the safe public status object
- `src/cli/status-dashboard.mjs`: writes status JSON files
- `dashboard/public/index.html`: static structure
- `dashboard/public/styles.css`: visual design and animation CSS
- `dashboard/public/app.js`: rendering, live refresh, route animation
- `test/dashboard-status.test.mjs`: data contract tests

`strategy.ethProfitability` is an execution-gated/operator-review summary. It may surface ETH-family candidate evidence and blockers, but it must not be treated as execution permission or live profitability approval.

`strategy.pivotPlan` powers the read-only `Pivot Watch` card. It may describe the next research lane and indicative capital floors, but it must not claim profitability, execution permission, or live readiness.

`strategy.strategySnapshot`, `prelive.connectedRefresh`, `prelive.connectedRefreshExecution`, `prelive.currentRoutePrelivePass`, `prelive.executionRunbook`, `prelive.exactRouteForkPackage`, `prelive.operationalJudgmentReview`, and `prelive.validation` are read-only operator/planning summaries. `strategy.strategyTracks.tracks[].kind` may now include `eth_family_loop`. That track is a research/review surface only and must remain compatible with `liveTrading: BLOCKED`.

`gateway.flowRoutes` must include aggregated route pairs for all Gateway routes, including non-BTC chain-to-chain pairs.

`gateway.assetCoverage` should make sampling bias visible. The dashboard should not imply unsupported assets just because recent successful quotes are concentrated in `wBTC.OFT`.

`gateway.announcedChainCoverage` compares the live Gateway API route inventory against BOB's public 11-chain messaging. As of the latest check, the live API source of truth has 10 chains and does not include Optimism or Sei, even though public BOB copy has mentioned both. Do not render missing announced chains as active nodes until they appear in `gateway.chains`; show them only as an API route gap.

`gateway.recentFlowEvents[].asset` and `gateway.flowRoutes[].assets[]` must include enough metadata for the UI to show what is moving:

- `ticker`
- `family`
- `icon`
- `src`
- `dst`

`gateway.recentFlowEvents[].segments[]` must describe each visible leg:

- source chain -> BOB Gateway uses the source asset icon
- BOB Gateway -> destination chain uses the destination asset icon

Do not animate one asset icon across the whole route when `src` and `dst` differ. That incorrectly makes native BTC look like it travels onto EVM chains.

## Route Visualization Rules

The visual model shows BOB Gateway as the central routing hub.

All visible movement should pass through `BOB Gateway`:

```text
Source chain -> BOB Gateway -> Destination chain
```

BOB Gateway and BOB Mainnet must be separate nodes. Do not use the BOB Mainnet node as the visual hub.

Required route classes:

- `btc_out`: Bitcoin is the source
- `btc_in`: Bitcoin is the destination
- `bob_out`: BOB is the source
- `bob_in`: BOB is the destination
- `chain_to_chain`: neither side is Bitcoin or BOB

Display rules:

- BTC outgoing paths and BTC incoming paths must use different colors.
- Route lines should be straight Gateway spokes. Do not use curved route lines unless the user asks for them again.
- Non-BTC chain-to-chain route animations must pass through BOB Gateway.
- All Gateway-supported chains must appear as nodes.
- Missing announced chains must not appear as route nodes until the Gateway API returns live routes for them.
- Chain nodes should show only the official logo and chain name. Do not add extra `source`, `gateway`, or `chain` labels.
- Recent traces should be limited to the newest records and show when and how much moved.
- Moving pulses should use official-domain asset icons where available: BTC, WBTC, USDC, USDT, ETH, PAXG, and XAUT. Unknown assets should fall back to a stable ticker badge.
- Chain logos and asset logos are different things. Do not reuse a chain logo to represent a token unless the token is the native asset of that chain.
- Native BTC should only appear on the Bitcoin <-> BOB Gateway leg. EVM-side BTC-like routes should use their actual wrapped ticker, for example `wBTC.OFT`, `WBTC`, `uniBTC`, or `solvBTC`.
- Bitcoin and Ethereum are anchor chains. Place them 180 degrees apart around BOB Gateway, with Bitcoin on the left side and Ethereum on the right side.
- Moving pulses should attach a small circular origin-chain badge, so the user can see which chain the route came from even after the pulse passes through BOB Gateway.

To avoid visual clutter, draw simple Gateway spokes to each chain instead of drawing every pair-to-pair edge at once. Pair movement should be shown by animated pulses traveling from source chain to BOB Gateway to destination chain.

## Chain Logos

The dashboard currently uses official-domain favicon URLs through Google favicon service:

```text
https://www.google.com/s2/favicons?domain=<official-domain>&sz=128
```

Domains are defined in `chainMeta` inside `dashboard/public/app.js`.

This was chosen because not every chain exposes a stable, easy-to-hotlink brand SVG. If replacing with local official assets later:

1. Put assets under `dashboard/public/assets/chains/`.
2. Keep filenames stable.
3. Update `chainMeta`.
4. Verify desktop and mobile screenshots.

Asset icons use the same official-domain favicon pattern for common assets. Domains are defined in `assetIconFor` inside `dashboard/public/app.js`. If replacing with local official token assets later:

1. Put assets under `dashboard/public/assets/tokens/`.
2. Keep filenames stable by ticker and asset family, for example `btc.png`, `wbtc.png`, `usdc.png`.
3. Update `assetIconFor`.
4. Verify that route pulses and recent traces still show conversion overlays correctly.

## Refresh Behavior

The browser reloads `dashboard-status.json` every 10 seconds with `cache: "no-store"` and a timestamp query string.

There should be no visible refresh button in the main experience.

The local automation should run the watcher and then regenerate dashboard status, so the browser can update without the user doing anything.

## Animation Behavior

The map uses:

- SVG route lines
- `.pulse` elements for moving dots
- `.trail` elements for fading afterimages
- CSS keyframes for smooth motion

The animation should remain decorative and informative. It must not imply confirmed execution or realized profit.

## Opportunity Watch

`Opportunity Watch` is read-only. It summarizes `data/gateway-scores.json` after `npm run score:gateway`.

It also surfaces DEX executable quote coverage from `data/dex-quotes.jsonl` and `data/dex-quote-failures.jsonl`. DEX quote coverage means "we asked a router for an executable token-to-stable quote"; it still does not mean the route is tradeable.

It may say that a review candidate exists, but it must not say that a route is tradeable or profitable. Candidate wording must remain manual-review oriented until live execution architecture, DEX executable quotes, exact gas estimation, UTXO-specific Bitcoin fee checks, and overfit gates are complete.

Bitcoin fee modeling now exists as a mempool.space estimate, but it is not exact wallet/UTXO execution cost. Do not imply that native Bitcoin routes are execution-ready just because `bitcoinFee.latest` is present.

The panel should stay plain:

- `No clean edge yet`
- `No score yet`
- `Review candidate found`

Avoid words like `profit`, `guaranteed`, `execute`, or `trade now`.

Expected runtime checks:

- `.chain-node` count should match supported chains.
- `.route-line` should be nonzero.
- `.pulse` and `.trail` should appear after a few seconds.
- Console should have zero errors.

## Mobile Layout

The dashboard is mobile-first.

Important mobile behavior:

- BOB Gateway stays visually central.
- BOB Mainnet appears as a normal outer chain node, not as the hub.
- Outer chain nodes remain aligned with their spoke endpoints.
- Text must not overflow its parent.
- The first screen should show the live map before supporting panels.

## Common Mistakes To Avoid

- Do not reintroduce a table-first dashboard.
- Do not put Cloudflare in the monitoring or execution path.
- Do not publish raw JSONL files.
- Do not let gas or audit data dominate the map.
- Do not show profitability unless it is based on measured, executable data.
- Do not treat an update alert as permission to trade.
- Do not let the dashboard itself hold keys, sign transactions, or decide to trade. `liveTrading: ALLOWED` is a status report from the gates, not permission granted by the dashboard.
